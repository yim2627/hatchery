# 아키텍처 규칙

앱 구조, 화면 구성, 책임 경계와 관련된 작업에 이 스킬을 적용한다.

## 핵심 원칙

- 현재 아키텍처 스타일({{ARCHITECTURE_STYLE}})을 유지한다.
- UI 컴포넌트는 프레젠테이션에만 집중한다.
- 비즈니스 로직을 UI 컴포넌트에 넣지 않는다.
- API·저장소 접근을 UI 컴포넌트에 넣지 않는다.
- 숨겨진 전역 상태보다 의존성 주입을 선호한다.
- 새 추상화를 만들기 전에 기존 패턴을 재사용한다.
- 이미 확립된 경우가 아니면 `Manager`, `Helper`, `Util` 같은 모호한 이름을 피한다.

## 레이어 책임

- **UI 레이어**: 상태 렌더링, 유저 인터랙션 전달, 최소한의 프레젠테이션 로직만 포함.
- **프레젠테이션 레이어**: UI 상태 준비, 비동기 작업 조율, 도메인 결과를 UI용 상태로 변환.
- **도메인 레이어**: 비즈니스 규칙 포함, 가능한 한 플랫폼 비의존적으로 유지.
- **데이터 레이어**: API / 스토리지 접근 조율.
- **인프라 레이어**: API 클라이언트, DB, OS 연동, 시스템 프레임워크.

## 경계

- 모든 네트워크 접근은 {{NETWORK_LAYER_NAME}}을 통해야 한다.
- 모든 저장소 접근은 {{PERSISTENCE_LAYER_NAME}}을 통해야 한다.
- 의존성 경계를 명확하게 유지한다.
- 명시적으로 요청받지 않는 한 새로운 아키텍처 패턴을 도입하지 않는다.
